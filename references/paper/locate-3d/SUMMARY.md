# Locate 3D — SoTA 3D referring-expression localization from raw RGB-D, powered by 3D-JEPA, a JEPA-style SSL that masks-and-predicts in latent space over point clouds featurized with frozen 2D foundation models

**Authors:** Sergio Arnaud*, Paul McVay* (joint first), Ada Martin†, Arjun Majumdar, Krishna Murthy Jatavallabhula, Phillip Thomas, Ruslan Partsey, Daniel Dugas, Abha Gejji, Alexander Sax, Vincent-Pierre Berges, Mikael Henaff, Ayush Jain, Ang Cao, Ishita Prasad, Mrinal Kalakrishnan, Michael Rabbat, Nicolas Ballas, Mido Assran, Oleksandr Maksymets, Aravind Rajeswaran‡, Franziska Meier‡ (joint last) — FAIR at Meta

## TL;DR
Locate 3D localizes an object in a 3D scene from a referring expression ("the small coffee table between the sofa and the lamp"), operating **directly on posed RGB-D sensor streams** (no ground-truth meshes, region proposals, or instance masks at inference). Its core SSL contribution is **3D-JEPA**: take a point cloud whose per-point features are **lifted from frozen 2D foundation models** (CLIP + DINOv2 + SAM masks), mask out spatial regions, and train an encoder–predictor to predict the **EMA-target encoder's latent features** of the masked regions (JEPA / I-JEPA recipe in 3D). The pretrained PointTransformer-v3 (PTv3) encoder is then fine-tuned with a language-conditioned decoder that jointly predicts 3D masks + bounding boxes. It sets SoTA on SR3D/NR3D/ScanRefer (joint Acc@25 **58.5%→61.7%**, and **63.7%** with the new dataset), with 3D-JEPA giving +1.9 in-domain (59.8→61.7) and +5.2 OOD (51.5→56.7), and deploys on a Spot robot (8/10 pick success).

## Problem & motivation
3D referential grounding (3D-RefExp) is split between two unsatisfying camps. (1) Specialist models trained on small benchmarks that **require clean meshes and/or ground-truth region proposals at inference** — undeployable on real devices. (2) Pipelines that bolt **2D VLMs/LLMs onto hand-crafted 3D representations** — strong language, weak 3D. Locate 3D targets the realistic deployment regime: input is just the sensor observation stream. Conceptually it argues that lifted 2D features are only **local** (per object/mask/patch), analogous to word embeddings; what 3D needs is **contextualized** scene-level features (analogous to BERT vs word2vec), which 3D-JEPA provides.

## Method
Three phases (Figure 1).

**Phase 1 — Preprocessing / feature lifting.** Unproject posed RGB-D to a point cloud; voxelize at **5 cm**. Per point, concatenate: dense **DINOv2** patch features; **CLIP** features computed **per-SAM-mask** then painted back onto the mask's pixels (CLIP is global, so SAM gives density; a learnable token fills pixels in no mask); plus a harmonic encoding of RGB. Lift 2D feature maps to 3D via trilinear-weighted averaging per voxel (ConceptFusion-style). Output: `PtC_lift = {(x,y,z, f_i)}`, `f_i ∈ R^d`. This high-dim featurized cloud (input dim = ViT feature dim) is the SSL input.

**Phase 2 — 3D-JEPA SSL.** Encoder `E_θ` and predictor `P_φ` trained by masked latent prediction:
`min_{θ,φ} || P_φ(E_θ(PtC̃), M) − sg( Ē_θ(PtC) ) ||`, where `PtC̃` is the masked view, `M` the mask variable, `sg` stop-grad, and `Ē_θ` the **EMA (exponential moving average) target encoder** that prevents collapse. Loss is per-point over masked points, averaged. Key design:
- **Latent-space (not input-space) prediction** is essential here: the input is high-dimensional lifted ViT features and directly reconstructing them is hard; teacher–student masked prediction (à la I-JEPA, DINOv2) works best.
- **Encoder = PTv3** (serializes the cloud via space-filling curves, groups + local attention, conv-like). **Predictor = transformer with block-sparse attention + rotary (Cartesian x/y/z) positional embeddings**, learnable mask tokens + global registers; sparse attention was crucial for throughput, memory, and stability. Sub-manifold sparse conv stem tokenizes points.
- **Masking pattern** matters a lot: **serialized percent-based masking** wins — (1) mask contiguous *regions* (spatially proximate points) not random points, to force spatial reasoning over local interpolation; (2) mask a *percentage* of the scene (e.g. ~8%) not a fixed point count, so it handles clouds from ~1k to ~200k points. Beats radius-based and fixed-count masking (validated by a frozen element-wise probe over pretraining).

**Phase 3 — Language-conditioned decoder.** Inputs: 3D-JEPA point features (projected to E=768, + learned 3D positional embeddings) and per-word CLIP text features; plus a set of learnable object queries (concatenated with language tokens = "queries"). **n=8 decoder blocks**, each doing 3 attentions: query self-attention, query→point cross-attention, and a **point→query cross-attention that updates the point features** (inspired by Jain et al. 2025). Three heads per query: a **mask head** (MLP·point-feature dot product → per-point logits, Mask2Former-style), a **text-alignment head** (MDETR-style alignment matrix grounding each query to a noun), and a **novel transformer box head** (concatenate projected xyz to point features, cross-attend with queries, MLP-regress the box). Joint mask+box supervision is the winning recipe.

**Training Locate 3D.** Decoder from scratch + fine-tune the 3D-JEPA encoder. Composite loss `L = λ_dice L_dice + λ_ce L_ce + λ_box L_box + λ_giou L_giou + λ_align L_align` (mask = dice + CE; box = L1 + generalized IoU; alignment = focal). Hungarian matching (DETR), deep supervision at every decoder layer, EMA of model weights for eval. To avoid destroying pretrained features: **stage-wise LR schedule** (LP-FT style) over 40 epochs — first 17 epochs freeze encoder / careful adaptation, then joint training with encoder LR = 0.5× decoder LR. **Supervise in sensor space** (transfer mesh labels onto noisy sensor points via k=8 NN) rather than mesh space, so the model must process sensor noise/artifacts instead of ignoring ~50% of points.

**Locate 3D Dataset (L3DD).** New human-annotated 3D-RefExp dataset over **three** scene sources — ScanNet, ScanNet++, ARKitScenes — with **131,641 annotations** across **1,346 scenes / 23,549 objects**. Provides human-generated masks AND grounded anchors AND multi-dataset coverage in one effort (prior datasets like ScanEnts3D / ARKitSceneRefer cover only 1 scene dataset and accumulated these annotation types over years). Roughly doubles training-annotation quantity and ~5× the size-adjusted venues with dense RefExp supervision; enables principled scene-generalization study. Model trained with it = **Locate 3D+**.

## Key results
All numbers are top-1 accuracy on validation, **without** ground-truth proposals; "Joint Eval" = SR3D+NR3D+ScanRefer combined.

- **Benchmark SoTA (Table 2).** Locate 3D, on **sensor point clouds**, beats the prior best (PQ3D, which uses *mesh* point clouds): Joint Acc@25 **61.7** (PQ3D 58.5), SR3D **65.8**, NR3D **53.7**, ScanRefer **59.9**. Locate 3D+ (with L3DD) pushes to Joint **63.7** / SR3D 68.2 / NR3D 56.1 / ScanRefer 61.1. Under the *same sensor-PC assumptions*, prior BUTD-DETR scores only 40.7 (Joint) → Locate 3D 61.7 is a +21 jump. VLM baselines (GPT-4o pipeline 38.6, Llama 28.8) trail badly.
- **3D-JEPA ablation (Table 3).** CF (CLIP+DINO) features >> RGB (53.9 vs 28.9 @25). Adding an encoder helps even randomly-init (53.9→59.8). **Frozen 3D-JEPA** beats raw CF (+3/+4 @25/@50). **Fine-tuned 3D-JEPA is best: 61.7 @25 / 49.4 @50** vs random-init encoder 59.8/47.0 — i.e. SSL pretraining + task fine-tuning is the winning combo.
- **2D-foundation-feature scaling (Table 4).** CLIP-L + DINOv2 (61.7) > CLIP-L alone (59.2) > CLIP-B/MobileSAM (53.7) ≈ DINOv2-only (53.7). Advances in 2D foundation models translate to 3D gains.
- **Contextualization probe.** Point-wise text-conditioned probe: 3D-JEPA 39% vs ConceptFusion 34% (description match); 73% vs 66% on relaxed noun-correctness — evidence the features are genuinely contextualized.
- **Decoder ablations (Table 6/7).** Mask+box+transformer box head 61.7 / 49.4; mask-only 55.4 (DBSCAN post-proc 58.4); box-only collapses (0.3); MLP box head 35.6. Larger decoder helps, and 3D-JEPA gives a stable +3–4% across all decoder sizes.
- **Generalization (Table 5/8).** Trained on ScanNet only, Locate 3D generalizes to L3DD splits: ScanNet++ **56.7**, ARKitScenes **46.2**, robot held-out env (FRE) 52.0 — CF features and 3D-JEPA each add large OOD gains (e.g. ARKitScenes 11.3 RGB → 41.7 CF → 46.2 +JEPA). Adding L3DD training data (Locate 3D+) jumps SN++ to **83.9**, ARKitScenes 57.6, FRE 73.5 (but in-domain L3DD data then dilutes 3D-JEPA's marginal benefit).
- **Robot deployment.** Spot robot, multi-room held-out apartment, "navigate + pick the plush toy": **8/10 success**, vs best baseline 5.66/10.
- **Compute.** With cached 2D features, a forward pass on a 100k-point scene is **~1 s / 8 GB VRAM on an A100**.

## Relevance to the EB-JEPA hackathon
This is the clearest **point-cloud JEPA** in the reference set — directly relevant to **Track 6 (point clouds)** — and maps cleanly onto the encoder / predictor / target / regularizer recipe:
- **It IS a JEPA (latent masked prediction, EMA target, stop-grad, no reconstruction).** Eq. 1 is literally I-JEPA's objective ported to 3D: context encoder `E_θ`, predictor `P_φ`, EMA target `Ē_θ`, loss in latent space over masked points. The collapse-avoidance mechanism is the **EMA target encoder** (asymmetric, like BYOL/I-JEPA), *not* an explicit variance/covariance regularizer — a useful contrast if EB-JEPA's angle is energy/anti-collapse losses.
- **Contrast with Point-JEPA (Saito et al., WACV 2025).** Point-JEPA is a *from-scratch* I-JEPA on **raw object-level** point clouds (it learns geometry-only representations from xyz, with a sequencer to order patches, ShapeNet-scale objects). Locate 3D's 3D-JEPA instead **(a) operates on scene-level clouds** (1k–200k points, not single objects), and **(b) does not learn from raw geometry — its per-point input features are frozen CLIP+DINO+SAM 2D-foundation features lifted into 3D**, so 3D-JEPA's job is to *contextualize/denoise/smooth* already-rich 2D-distilled features rather than learn semantics from scratch. The paper explicitly notes a naive masked-modeling instantiation fails at this scale/feature-dim, motivating latent prediction + serialized percent masking. Net: Point-JEPA = "JEPA learns the features"; 3D-JEPA = "JEPA contextualizes lifted 2D-foundation features."
- **Transferable design choices for a point-cloud JEPA:** serialized percent-based region masking (vs radius/fixed-count); PTv3 (space-filling-curve serialization) encoder + block-sparse + rotary-on-Cartesian-coords transformer predictor; latent-space prediction being *necessary* when inputs are high-dim lifted ViT features; LP-FT-style stage-wise fine-tuning to preserve SSL features; sensor-space (noisy) supervision.
- **Downstream framing:** demonstrates a JEPA-pretrained 3D encoder fine-tuned for a *language-conditioned* localization task with a DETR/Mask2Former/MDETR-style decoder — a template for "JEPA backbone + task head" beyond the world-model/planning tracks.

## Caveats / open threads
- **3D-JEPA does not learn from raw geometry** — it is bootstrapped entirely on frozen 2D foundation features; its gains over raw CF features are real but modest in-domain (+1.9 @25) and larger OOD (+5.2). Whether the JEPA stage would help on *RGB-only* clouds is not the headline setting (the RGB+random-encoder line is 51.4, far below CF).
- **Static-scene assumption.** Caching 2D features (offline for ScanNet, during exploration for the robot) is what makes inference cheap; dynamic scenes would need real-time featurization + continuous map updates (cited as open, e.g. Lifelong LERF).
- **Heavy frozen-foundation-model stack** (CLIP-L + DINOv2 + SAM-H) in the loop — the "3D" model inherits their cost and biases; results scale with 2D-model quality.
- **L3DD dilutes the SSL benefit:** once in-domain L3DD training data is added (Locate 3D+), 3D-JEPA's marginal contribution on L3DD evals shrinks — the SSL win is largest in the low-in-domain-data / OOD regime.
- **No public code/weights** noted in the paper; preprint, not yet peer-reviewed. Robot eval is a single apartment, 10 trials.
- Box-only supervision fully collapses (0.3%) — the dense **mask** signal is what carries learning; box head only sharpens localization on top of it.
